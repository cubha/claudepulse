> 생성일: 2026-04-30
> 프로젝트: 코드베이스 가시화 IDE 익스텐션 시스템 (작업명 codebase-viz-extension)
> 리서치 깊이: 기본 + ux-researcher (사용자 명시 핵심 요구로 추가)
> 참여 에이전트: 4명 (tech-researcher, architecture-researcher, market-researcher, ux-researcher) + 팀리드 Context7

## 1. 프로젝트 개요

**목적**: **비개발자(PM/디자이너/기획자/대표)도 5분 안에 읽을 수 있는** 프로젝트 구조 시각화 IDE 익스텐션. 단, 가독성을 위해 필수 정보 누락 금지.

**타겟 사용자**: 1차 — PM/디자이너/기획자. 2차 — 신규 합류 개발자(Onboarding), 시니어 개발자(아키텍처 검토).

**핵심 정보 요구 3축**:
1. 렌더링 아키텍처 (SSR/CSR/SSG, 라우팅, 데이터 페칭 흐름)
2. 화면별 디렉토리/페이지 구조 — 영역-컴포넌트 매핑
3. DB ERD + 스키마 + 화면-테이블 매핑

**출력 매체**:
- IDE 익스텐션 (VSCode + JetBrains)
- PIP 또는 터미널 패널 (IDE 내부)
- Multi-format Export: PPT, Notion, Obsidian, Markdown

**워크스페이스 컨텍스트**: 다국어·모노레포·다중 서비스 혼재 (Next.js, Nuxt, Vue, Vite, FastAPI, NestJS 등).

**컨텍스트 분리 규약**: 기존 Fallback 구조(Graphify+dependency-cruiser+Mermaid 3-Layer)는 별도 메모리(`project_codebase_viz_fallback_temp.md`)로 격리. 본 보고서와 혼합 제안 금지.

## 3. 아키텍처 패턴 리서치 (architecture-researcher)

### 3-1. 분석 코어 vs IDE 클라이언트 분리

**1차 권장**: **LSP-like 분리 (외부 코어 프로세스 + JSON-RPC)**.
- 코어 언어: Node.js 1차 (다국어 파서 생태계 풍부, OpenAPI/Tree-sitter 바인딩 풍부)
- 성능 병목 모듈만 선택적 Rust 바인딩
- VSCode + JetBrains 양쪽 지원에 결정적

**리스크**: 두 IDE protocol surface 유지 → `$/`-prefix LSP 확장 메시지로 capability 협상.

### 3-2. 정적 분석 IR — 2-tier 설계

**1차 권장**: 2-tier IR
- **Core Symbol Layer**: SCIP 호환 그래프 (Document/Symbol/Occurrence/Relationship) — 언어 의미 인덱싱
- **Domain Layer**: Route, Page, Component, ApiHandler, Service, Model, Table 노드 + renders/calls/queries/imports 엣지
- **Adapter Registry**: 프레임워크별(Next.js/Nuxt/Vue/Vite/FastAPI/NestJS) 추출기 — Tree-sitter Query + 프레임워크 메타(OpenAPI/decorators) 결합

**근거**: SCIP는 LSIF 대비 human-readable string ID + 4–8x 작은 인덱스 + cross-repo 합성 유리.

**비개발자 가독성 핵심**: Domain Layer를 **요약 IR**(Page-API-Table 거시 노드만) + **상세 IR**(Component/Service까지) 2단으로 직렬화.

### 3-3. IDE 뷰 표시 패턴 (PIP/패널/Webview)

**1차 권장**: **Webview 단일 React 번들 + 3 호스트 (VSCode WebviewPanel, VSCode Floating Window, JetBrains JCEF Tool Window)**.
- 메시지 채널: JSON-RPC 스타일 추상화 (postMessage / acquireVsCodeApi ↔ JBCefJSQuery)
- PIP는 1차로 Floating Editor Window — "always-on-top"이 진정 필요한 경우만 OS 외부창 옵션

**리스크**:
- Floating window 이동 시 reload → IR/뷰 상태는 **코어가 source of truth**, 리하이드레이션 메시지 프로토콜 필수
- `retainContextWhenHidden`은 메모리 부담 큼 — 큰 그래프엔 부적합, getState/setState 우선
- "VSCode PIP API"는 사실상 없음 — 표기는 "Floating Window" 또는 "보조 창"으로 정확히

### 3-4. 증분·캐시 전략

**1차 권장**: **2층 캐시**
- **패키지 단위 task-level 캐시**: `package.json + lockfile + 소스 해시 + 분석기 버전`을 키로 IR 산출물 캐시 (Turborepo 콘텐츠 해시 모델 응용)
- **세션 단위 incremental**: Tree-sitter incremental tree로 편집 파일만 재파싱

**저장**: SQLite (IR 그래프 + FTS5 인덱스), 위치는 IDE workspace storage(격리) + 큰 산출물은 `~/.cache/<extension>/<repo-hash>/`.

**리스크**: 캐시 무효화 키에 분석기 버전·어댑터 스키마 버전 포함 필수 (stale ghost IR 방지).

### 3-5. Export 파이프라인

**1차 권장**: **IR(요약/상세) → Layout → 포맷별 Renderer Adapter** 3단 파이프라인.
- 캐시: render 입력 해시(노드 부분그래프 + 스타일 + 레이아웃 시드) 기준 SVG 캐시
- **SVG 마스터** → PNG/PPT 임베드 fallback 항상 보유
- v1: **단방향 export**. Notion/Obsidian 양방향 sync는 v2로 보류 — IR 정합성 보호

## 5. UI/UX 트렌드 (ux-researcher)

> 비개발자 가정: 임원/대표는 Overview 1장, PM/기획자는 Drilldown 1단계, 디자이너는 화면-컴포넌트 매핑까지.

### 5-1. 정보 계층화 모델

**1차 권장**: **C4 L1+L2 + 4+1 Scenarios View**.
- L1(System Context): 임원/대표 1장 보고
- L2(Container): PM/기획자 의사결정
- L3(Component): 신규 개발자 온보딩
- L4(Code): 시니어 검토 — **비개발자 노출 금지**

**근거**: C4 공식이 L1·L2를 "non-technical audience" 청중으로 명시. 4+1의 Scenarios View는 Logical View 검증·트레이서로 비개발자에 부합.

### 5-2. 화면-컴포넌트 매핑

**1차 권장**: **와이어프레임 위 영역 라벨(1차) → 클릭 시 컴포넌트 카드(2차) → Storybook Docs 링크(3차)**.

**비개발자 함정**:
- Treemap을 1차 화면으로 쓰지 마라 (NN/g: area는 preattentive 아님)
- 파일명(`UserCard.tsx`)을 영역명과 동시 노출 금지
- Progressive Disclosure 필수

### 5-3. DB-화면 데이터 매핑

**1차 권장**: **(a) Conceptual ERD 1장 + (b) 화면별 필드 매핑 표 + (c) Business Glossary 평문 정의** 3종 세트.
- Crow's Foot 표기는 비개발자에 격리 — 한국어 라벨("1명-여러 주문")로 대체
- Glossary가 의미를, Dictionary가 위치를 — Atlan/Alation 권고
- ERD에 모든 테이블(로그/세션/오딧) 포함 금지 — 핵심 5~9개 한정 (Miller's Law)

### 5-4. 렌더링 아키텍처 비개발자 설명

**1차 권장**: **비유 인포그래픽 1장 + 결과 비교표(속도/SEO/실시간성)**.
- SSR=식당 즉석 조리, SSG=뷔페 미리 차림, CSR=재료 키트 배달 후 집에서 조리
- 시퀀스 다이어그램(UML)은 비개발자 채널에서 **제거** (IcePanel: alt/par 프래그먼트 학습 비용)

### 5-5. Multi-format Export 정보 손실 최소화

**1차 권장**: **단일 IR + 포맷별 Depth 매트릭스**.
- **PPT**: Depth 1~2 (1주제=1슬라이드 강제)
- **Notion**: Depth 무제한 (Toggle 중첩, Callout은 judiciously)
- **Obsidian**: Frontmatter (level/audience/area) + MOC 진입점
- **Markdown+Mermaid**: 노드 ≤50(고밀도) / ≤100(저밀도). 초과 시 자동 분할

### 5-6. UX 가이드라인 — 7원칙

1. Overview 1장이 첫 화면
2. Progressive Disclosure를 기본 인터랙션으로
3. 면적·각도는 의사결정에 쓰지 마라 (Treemap 정밀 비교 X)
4. DB는 Conceptual + Glossary로, 화면 매핑은 표로
5. 메커니즘이 아니라 결과 차이를 설명 (시퀀스 다이어그램 비개발자 제거)
6. 포맷별 Depth Matrix를 IR에 박아라
7. 용어는 한 곳에서 한 번만 정의 (Business Glossary가 SSOT)

> 출처: NN/g, C4 model, ArchiMate spec, 4+1 View Model(Kruchten arXiv), Storybook Docs, Figma Dev Mode, Atlan/Alation, IcePanel 등 23개 1차 URL.

## 7. 종합 분석 및 권장안

### 7-1. 크로스 분석 결과 (4명 결과 정합성)

| 매트릭스 | 정합 | 모순 |
|---|---|---|
| tech ↔ architecture | ts-morph+Tree-sitter ↔ SCIP+Domain 2-tier IR / Atlas+tbls ↔ Domain Model·Table 노드 / ReactFlow+ELK ↔ Webview 단일 번들 / dagre deprecated ↔ Layout 분리 | 없음 |
| tech ↔ market | Mermaid 허브 ↔ ChartDB Notion 임베드 패턴 / mermaid-cli+pptxgenjs ↔ 시장 PPT 1급 부재 갭 | 없음 |
| architecture ↔ market | SVG 마스터 단방향 Export ↔ 시장 PPT 1급 부재 / OSS+Pro 분리 가능 ↔ 시장 검증 모델 | 없음 |
| ux ↔ architecture | 요약/상세 2-tier 노출 ↔ 요약/상세 IR 2단 직렬화 / Depth Matrix ↔ 단방향 Renderer 어댑터 | 없음 |
| ux ↔ tech | Mermaid 노드 ≤50 ↔ ReactFlow+ELK 큰 그래프 분기 / Conceptual ERD ↔ Atlas 코드모델+tbls DB 진실 | 없음 |
| ux ↔ market | Conceptual ERD 권장 ↔ ChartDB ★5 가독성 정합 / Progressive Disclosure ↔ CodeSee onboarding tour | 없음 |

**결론**: **4명 결과 사이에 결정적 모순 없음.** 모두 "비개발자 가독성 = 요약/상세 2-tier + Progressive Disclosure", "VSCode PIP 정식 미지원 → Floating Window 차선책", "단방향 Export + SVG 마스터", "다국어 모노레포 = 멀티 파서 + Adapter Registry"에 합의.

### 7-2. 1순위 통합 권장안

**시스템 형태**
- VSCode 익스텐션 + JetBrains 플러그인. **단일 React Webview 번들**을 VSCode WebviewPanel·JetBrains JCEF Tool Window 양쪽에서 재사용.
- 분석은 Node.js sidecar 별도 프로세스 + JSON-RPC (LSP-like).

**파서/추출**
- 라우트: 파일시스템 컨벤션 스캔(1차) + 빌드 manifest(2차 검증)
- 컴포넌트: ts-morph + @vue/compiler-sfc + svelte/compiler + Tree-sitter (다국어 백엔드)
- DB: Atlas (다중 ORM HCL 통합) + tbls (DB introspection 진실원본)
- **화면-테이블 매핑**: 자동 통합 도구 시장 부재 → 직접 구현. ts-morph로 ORM 호출 정적 추적 + 백엔드 라우트→ORM→테이블 그래프 합성

**IR**
- 2-tier: SCIP 호환 Core Symbol Layer + Domain Layer(Route/Page/Component/ApiHandler/Service/Model/Table)
- 직렬화: 요약 IR(거시 노드만) + 상세 IR(미시) 동시 산출 — UX의 Progressive Disclosure / Depth Matrix와 정합
- Adapter Registry: Next/Nuxt/Vue/SvelteKit/Vite/FastAPI/NestJS 어댑터 + Tree-sitter Query

**캐시·저장**
- 2층: 패키지 task-level 콘텐츠 해시(Turborepo식) + 세션 incremental(Tree-sitter)
- 영속: SQLite + FTS5
- 위치: IDE workspace storage(격리) + 옵션 `~/.cache/<extension>/<repo-hash>/`
- 무효화 키에 분석기 버전·어댑터 스키마 버전 포함 필수

**UI 다이어그램**
- 인터랙티브 PIP/패널: **ReactFlow + ELK.js** (DOM 노드, 비개발자 친화 라벨/아이콘)
- 정적 export 허브: **Mermaid** (erDiagram, C4Context, C4Deployment 네이티브)
- 대규모(수천+ 노드): Cytoscape.js 보조

**Export 파이프라인**
- IR(요약/상세) → Layout → Renderer Adapter 3단
- **SVG 마스터** → PNG/PPT(pptxgenjs)/Notion/Obsidian
- v1: 단방향. Notion/Obsidian 양방향 sync는 v2
- 포맷별 Depth Matrix:
  - PPT(pptxgenjs): Depth 1~2, 1주제=1슬라이드
  - Notion(@notionhq/client + 2026-03-11 Markdown API): Depth 무제한, 토글 중첩
  - Obsidian: vault MD 직접 + frontmatter(level/audience/area) + MOC
  - Markdown+Mermaid: 노드 ≤50, 초과 시 자동 분할

**UX 원칙**
- C4 L1+L2 + 4+1 Scenarios View. L3 이하 비개발자 격리
- Progressive Disclosure 기본 인터랙션
- DB: Conceptual ERD + 필드 매핑 표 + Business Glossary 3종
- 렌더링: 비유 인포그래픽 + 결과 비교표 (시퀀스 다이어그램 비개발자 제거)
- Format Depth Matrix를 IR에 박기

### 7-3. 차별화 전략

- **3축 자동 매핑** (렌더링/화면-컴포넌트/DB-화면) — 시장 부재 영역
