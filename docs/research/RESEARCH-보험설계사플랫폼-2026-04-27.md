# 팀 리서치 보고서 — 보험설계사·소개인용 플랫폼 (가칭 InsureAgent)

> 생성일: 2026-04-27
> 프로젝트: 한국 등록 보험설계사·소개(모집)인용 영업 보조 SaaS
> 리서치 깊이: 기본 (4 에이전트)
> 참여 에이전트: tech-researcher, architecture-researcher, market-researcher, 팀 리더(규제·라이선스)

---

## 0. TL;DR — 한 줄로

> **개인 개발자가 만들 수 있는 가장 현실적 형태는 "GA·보험사 중립의 설계사 종합 업무 SaaS"다.** 마이데이터 직접 연동·청약 직접 처리는 처음부터 빼고, **고객 PDF 업로드 + OCR + 멀티 보험사 코드 관리 + 분급제 정산 시뮬레이터**를 무기로 잡는다. **MVP는 풀타임 6~8주, Claude Code+매일 2~3h(주말 포함) 시나리오 5~7주에 출시 가능, 월 인프라비 약 8만원**. 결제·소개인·OCR 등 무거운 모듈은 모두 MVP 이후로 미룬다. 토스 보험파트너 종료(2025.12)로 10만 설계사가 대체 도구를 찾고 있어 진입 타이밍이 좋다.

## 2. 4-TIER 시스템 분류 ★ 본 보고서의 핵심

> **수정 원칙**: MVP는 "정말 없으면 못 쓰는 것"만 남기고, 나머지는 단계별로 미룬다. 베타 → 첫 결제 → 차별화 순서.

### TIER 1. 필수시스템 (MVP — 0~9주, 베타 무료)

> 무료 베타에서 30명이 "써볼만하다"고 느낄 최소 기능. 결제·본인확인·고급 보안은 모두 빠진다.

| 시스템 | 핵심 기능 | 구현 방식 | 외부 비용 |
|---|---|---|---|
| **회원·인증** | 이메일 + 카카오/구글 소셜 로그인. **베타 단계: 본인확인 생략** | Supabase Auth | $0 |
| **단순 RLS (1계층)** | `agent_id` 기반 자기 데이터만. `ga_id`는 nullable, 멀티테넌트는 후순위 | Supabase RLS 기본 정책 | $0 |
| **고객 CRM** | 고객 등록·검색·태그·메모·가족관계. PII 별도 테이블 분리는 후순위 (Supabase 기본 at-rest 암호화로 출발) | 100% 자체 구현 | $0 |
| **계약 등록 (수동)** | 보험사·상품·증권번호·만기·보장내역 직접 입력, PDF/이미지 첨부 | 100% 자체 구현 | $0 |
| **만기·기념일 알림 (이메일)** | 생일·기념일·계약 만기 알림. SMS·알림톡은 후순위 | cron + Resend(무료 3K통/월) | $0 |
| **디지털 명함·공유 페이지** | 설계사 프로필 URL, QR, 카카오 공유 메타태그 | Next.js 페이지 | $0 |
| **기초 보장분석 계산기** | 사용자 입력 기반 보장 적정성 점수, 부족액 시뮬레이션 | 자체 알고리즘 | $0 |
| **개인정보처리방침·약관** | 개인정보보호위원회 표준 템플릿 활용. 변호사 자문은 첫 결제 전까지 미룸 | 무료 표준 템플릿 | $0 |
| **기본 운영 로깅** | Supabase 기본 로그 + Sentry 무료(5K 에러) | Sentry Free | $0 |

**MVP 월 운영비 합계**: **약 80,000원**
- Vercel Pro 27.6K + Supabase Pro 34.5K + 도메인 0.2K + Sentry/Resend/PostHog 무료 + 카카오·OpenAI 무료 또는 미사용
- 사업자등록·세무대리·변호사 자문은 **MVP 단계에서는 불필요**

> ⚠️ **MVP에서 의도적으로 뺀 것**(이전 보고서에 잘못 포함됐던 항목):
> - ❌ 본인확인(포트원 월 5만) → 결제 도입 시점에 추가
> - ❌ pgcrypto PII 분리 + pgaudit 5년 감사로그 → 데이터 5만건+ 누적 시점에 추가
> - ❌ 변호사 1회 자문 30~50만 → 첫 유료 결제·소개인 기능 도입 시점에
> - ❌ 3계층 멀티테넌트 RLS → 첫 GA 도입 시점에 ga_id 활용 시작
> - ❌ 사업자등록·세무대리 → 결제 도입 시점에

### 3-B. architecture-researcher: 권장 아키텍처 핵심 (요약)

#### 스택
- **MVP**: Next.js 14 (App Router) + Supabase **ap-northeast-2(서울)** + Vercel
- **인증**: Supabase Auth + JWT Custom Claims `{ga_id, agent_id, role}`
- **DB**: PostgreSQL + RLS(3계층) + pgcrypto(PII) + pgaudit(감사)
- **모바일**: PWA 우선 → Expo 유니버설(웹+앱) 마이그레이션
- **오프라인**: PowerSync (Supabase 공식 통합, LWW 충분)
- **외부 API**: 어댑터 패턴 + Outbox 큐 (마이데이터 등 미래 연동 대비)

#### 핵심 데이터 모델

```sql
ga_organizations (id, name, ...)
agents           (id, user_id→auth.users, ga_id→ga_orgs, role)
customers        (id, agent_id, ga_id, ...)        -- RLS 적용
customer_pii     (id→customers, name_enc, rrn_enc, phone_enc)  -- 별도, pgcrypto
contracts        (id, customer_id, insurer, product, ...)
audit_log        (immutable, 5년 보관)
```

#### RLS 정책 (3계층 핵심)
- 설계사 본인: 자기 고객만 SELECT
- GA 매니저: 소속 GA 전체 고객 SELECT
- PII 테이블: Security Definer 함수로만 접근

#### 디렉토리 (요약)
```
app/                  # 라우팅 (멀티테넌트 [ga_slug]/...)
features/             # 도메인별(고객·계약·견적·정산)
lib/adapters/         # 본인확인·OCR·마이데이터 어댑터
supabase/migrations/  # SQL
```

전체 패턴·SQL 예시는 부록 §6 참조.

### 4-D. 개인 시나리오 환산표 (Claude Code + 부분 시간)

> **유효시간 비율 × CC 가속 배수 × 컨텍스트 손실(0.85) = 풀타임 대비 진척률**

| 시나리오 | 주당 시간 | CC 배수 | 진척률 | Phase 1 | Phase 2(코드) | Phase 3 |
|---|---|---|---|---|---|---|
| 평일만 2h | 10h | 2.0배 | **0.56배** | 11~14주 | 14~18주 | 28~38주 |
| **평일 2.5h + 주말 4h × 2** ★ | 20.5h | 2.0배 | **1.16배** | **5~7주** | **7~9주** | **14~19주** |
| 평일 3h + 주말 5h × 2 | 25h | 2.5배 | **1.76배** | 4~5주 | 5~6주 | 10~13주 |

#### 핵심 가정 검증
- **CC 가속 배수 ≈ 2.0배** (MVP는 보일러플레이트 30%·CRUD 25% 비중 → 가중평균 2배 도출)
- **컨텍스트 손실 15%**: 매일 시작 시 어제 코드 파악·다음 작업 결정에 첫 20분 손실
- 풀타임 기준 실코딩 30h/주 가정

#### 단축 불가 영역 (코드 시간이 아닌 달력 시간)

| 항목 | 대기 시간 | 영향 Phase |
|---|---|---|
| 토스페이먼츠 가입 심사 | 1~2주 | Phase 2 |
| 카카오 비즈채널 + 알림톡 템플릿 심사 | 1~2주 | Phase 2 |
| 변호사 1회 자문 일정 | 3~7일 | Phase 2 |
| Apple 앱 첫 심사 | 1~2주 | Phase 3 |
| 베타 사용자 피드백 수집 | 2~4주 | Phase 1~2 |
| ISMS-P / 마이데이터 협상 | 수개월 | Phase 4 |

→ **Phase 2부터는 코드 단축 효과를 외부 대기가 상쇄**한다. 따라서:

```
누적 일정 (개인 시나리오 ★):
  MVP 출시        ≈ 6~8주   (Phase 0+1)
  첫 유료 결제    ≈ 14~19주 (3.5~5개월) ← Phase 2 외부대기 +2~4주 포함
  GA B2B 진입     ≈ 28~38주 (7~9개월)
```

#### 함정 (긍정 편향 주의)
- "매일 꾸준히 2~3h"가 가장 어려운 부분. 야근·약속·번아웃으로 평균 1.5h가 현실
- CC 가속은 잘 시킬 줄 알아야 2배. 모호하게 시키면 오히려 느려짐
- 분급제 시뮬레이터처럼 **공식 데이터가 비공개인 영역은 CC가 도와주지 못함**
- 사용자 피드백 사이클은 단축 불가 — 베타 30명 모으는 데 2~4주 그냥 흐른다
- 결정 피로(BM·디자인·우선순위)는 본인 몫. 결정 미루면 코드 시간만 늘어남
