## 1. SubTask 정의

### MVP-1: jsonl 증분 파서 + cwd 우선 디코더

**목표**: `~/.claude/projects/<encoded>/<sessionId>.jsonl`을 mtime+offset 기준 증분 파싱하고 `message.id`로 dedup한 `UsageRecord[]`를 반환.

**파일**:
- Edit `src/services/JsonlParser.ts` (`parseIncremental`, `toUsageRecord`)
- Edit `src/utils/pathDecoder.ts` (jsonl 첫 user 라인의 `cwd` 우선 채택, 폴백으로 단순 역변환)

**구현 포인트**:
1. `fs.createReadStream(path, { start: byteOffset, encoding: 'utf8' })` + readline
2. 라인별 `try/catch JSON.parse` — invalid 줄 silent skip
3. `entry.type === 'assistant' && entry.message?.usage` 필터
4. `this.seenMessageIds.add(message.id)` — 이미 있으면 skip (🔴 누락 시 billing 불일치)
5. cache 토큰 분리: `cache_creation_input_tokens` / `cache_read_input_tokens`
6. `pricing.calculateCost`로 USD 산출 (jsonl `costUSD`는 cross-check만)
7. 새 `byteOffset` = 마지막 라인 끝 누적 바이트 수 (UTF-8 길이 주의 → `Buffer.byteLength`)

**DoD**:
- [ ] 동일 `message.id` 5번 들어와도 records 길이 1
- [ ] `byteOffset` 재호출 시 처음부터 재파싱 안 함 (records 길이 0)
- [ ] cache_creation/cache_read 합산이 input과 분리되어 `UsageRecord`에 보존
- [ ] 비ASCII 문자 포함 라인에서 offset drift 없음

**의존**: 없음 (Wave 1)

**검증**:
- vitest `JsonlParser.test.ts` — 합성 fixture 3종 (단일·dup·증분)
- 실제 `~/.claude/projects/<현재 워크스페이스 인코딩>/*.jsonl` 1개 수동 파싱

### MVP-2: chokidar 와처 — .jsonl 필터 + 초기 스캔 + 디바운스

**목표**: `~/.claude/projects/**/*.jsonl` 변경 이벤트를 디바운스하여 onChange 콜백으로 흘려보낸다. 초기 1회 전체 스캔도 포함.

**파일**:
- Edit `src/services/FileWatcher.ts`

**구현 포인트**:
1. `chokidar.watch(rootPath, { ignored: /(^|[/\\])\../, ignoreInitial: false, depth: 4 })` — 초기 add 이벤트도 받기
2. `.jsonl` 외 확장자는 `schedule()` 진입 전에 차단 (`if (!path.endsWith('.jsonl')) return`)
3. 디바운스: 마지막 이벤트로부터 `debounceMs`(기본 300ms) 후 1회만 onChange 호출
4. `add` / `change` / `unlink` 모두 처리 (unlink 시도 트리거)
5. `dispose()`에서 timer + watcher 모두 정리 (현재 stub OK)

**DoD**:
- [ ] 초기 시작 시 모든 기존 `.jsonl`에 대해 1회 onChange 발생
- [ ] 100ms 간격으로 5번 이벤트 발생 시 onChange는 1회만 호출
- [ ] 비-jsonl 파일 변경(예: `~/.claude/projects/.gitkeep`) 무시

**의존**: 없음 (Wave 1) — JsonlParser 호출은 orchestrator(MVP-4) 책임

**검증**: vitest `FileWatcher.test.ts` — temp dir에 fixture 생성 + 디바운스 카운트

### MVP-3: UsageAggregator — today / MTD / 5h Window / topProjects

**목표**: `UsageRecord[]`을 받아 `SnapshotPayload`(today·MTD·topProjects·billingWindow) 산출.

**파일**:
- Edit `src/services/UsageAggregator.ts`

**구현 포인트**:
1. `ingest()` 시 records를 보관하되 `messageId` 키 Map으로 멱등 보장 (parser 외 추가 dedup 방어)
2. `today` = `now`의 캘린더 일자 필터 → `AggregatedStats`
3. `monthToDate` = `now.getMonth()` 일자 필터
4. `topProjects` = projectPath별 group → `cost desc` top 5 → `ProjectSummary`
5. **5h Window** (UX-BRIEF + 디자인 5h 카드 기준):
   - `windowStart` = 가장 오래된 *최근 5h 내* record의 timestamp
   - `windowEnd` = `windowStart + 5h`
   - `pctTimeRemaining` = `(windowEnd - now) / 5h * 100` (시간 기준 100% 정확)
   - `tokensInWindow` = window 안 records 토큰 합
   - `estimatedPctRemaining` = (선택) — Anthropic 한도 비공개라 V1은 undefined로 두고 Sidebar에서 "—%" 표시 가능

**DoD**:
- [ ] 빈 records → snapshot이 모두 0 / undefined로 안전
- [ ] today 필터가 자정 경계에서 정확히 분리
- [ ] 5h Window가 `now - 5h` 이전 record는 제외
- [ ] topProjects 정렬·top 5 cap

**의존**: MVP-1 (UsageRecord 형태 확정 후) → Wave 2

**검증**: vitest `UsageAggregator.test.ts` — 합성 records로 today/MTD/5h 경계 케이스

### MVP-4: Orchestrator wire + StatusBar 실데이터 + messenger handler

**목표**: FileWatcher → JsonlParser(증분) → CacheStore(file-index) → UsageAggregator → SnapshotPayload → (a) StatusBar 갱신 (b) messenger `PushSnapshot` 브로드캐스트. **데이터 파이프라인 첫 통합 지점.**

**파일**:
- Edit `src/extension.ts` (activate 안에서 orchestrator 구성)
- Write `src/messaging/handlers.ts` (Messenger 인스턴스 + GetSnapshot/RequestRefresh 등록)
- Edit `src/providers/StatusBarController.ts` (필요 시 `loading` 상태 보강 — 현재 update 시그니처는 충분)
- Edit `src/services/CacheStore.ts` (필요 시 — 기존 fs API로 충분, **수정 가능성 낮음**)

**구현 포인트**:
1. activate 진입 시:
   - `cacheStore.loadFileIndex()` → 기존 offset 복원
   - `FileWatcher` 생성 + `onChange` = `runIncrementalScan()`
2. `runIncrementalScan()` 시퀀스:
   - 변경된 `.jsonl` enumerate (chokidar는 path를 콜백으로 안 줌 → onChange 시 root 전체 mtime 체크 또는 watcher event에서 path 누적 큐 도입)
   - 각 파일에 대해 `JsonlParser.parseIncremental(path, fileIndex[path]?.byteOffset ?? 0)` 호출
   - file-index 갱신 → `cacheStore.saveFileIndex`
   - `aggregator.ingest(records)` → `aggregator.buildSnapshot()` → snapshot
3. snapshot 적용:
   - `statusBar.update({ tokensToday: snapshot.today.totalTokens, sessionCostUsd: snapshot.today.cost })`
   - `messenger.sendNotification(PushSnapshot, BroadcastType, snapshot)` (Sidebar/Panel webview 모두에)
4. `Messenger.onRequest(GetSnapshot, ...)` → 마지막 snapshot 반환
5. `Messenger.onNotification(RequestRefresh, ...)` → `runIncrementalScan()` 재호출
6. `claudepulse.refresh` 커맨드 → `runIncrementalScan()`

**DoD**:
- [ ] F5 후 StatusBar에 실제 today 토큰 수·USD 표시 (placeholder `0`/`$0.00` 아님)
- [ ] `~/.claude/projects/`에 새 jsonl 라인 추가 시 300ms 내 StatusBar 갱신
- [ ] Output 채널 `Claudepulse`에 `[info] incremental scan: N files, M new records` 로그
- [ ] 사이드바 새로고침 버튼(추후 MVP-5에서 추가) 호출 → snapshot 재계산

**의존**: MVP-1·2·3 모두 완료 후 (Wave 3) — **단일 통합 지점**

**검증**:
- 수동: F5 후 `~/.claude/projects/.../*.jsonl`에 한 줄 echo로 추가 → StatusBar 갱신 관찰
- `verify.sh` 통과
